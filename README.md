# Blog Platform API

REST API для блог-платформы с поддержкой ролевой модели, JWT-аутентификации, пагинации, фильтрации и сортировки.

---

## Описание проекта

Проект реализует backend-часть блог-платформы, предоставляющей следующий функционал:

- **Регистрация и аутентификация** пользователей через JWT Bearer Token.
- **Ролевая модель**: Reader (читатель), Author (автор), Moderator (модератор) — с разграничением прав доступа к каждому эндпоинту.
- **CRUD-операции** для статей, комментариев, категорий и пользователей.
- **Лайки** — аутентифицированный пользователь может лайкнуть/убрать лайк со статьи.
- **Расширенный листинг статей**: пагинация, фильтрация по категории и статусу, сортировка по популярности (количеству лайков).
- **Автодокументация** через Swagger UI (`/docs`) и ReDoc (`/redoc`).
- **Логирование** в двух дополняющих друг друга слоях: оперативные логи через стандартный модуль `logging` (вывод в stdout — для мониторинга/дебага в реальном времени) и **постоянная история действий** в таблице БД `action_logs` — доступна через `GET /api/logs` и переживает перезапуск контейнера.

## Запуск проекта

### Вариант 1 — локально (без Docker)

```bash
# 1. Клонировать репозиторий
git clone <repo-url>
cd blog_api

# 2. Создать виртуальное окружение
python -m venv .venv
source .venv/bin/activate      # Linux / macOS
# .venv\Scripts\activate       # Windows

# 3. Установить зависимости
pip install -r requirements.txt

# 4. Скопировать конфигурацию
cp .env.example .env
# При необходимости отредактировать .env

# 5. Заполнить БД начальными данными
python seed.py

# 6. Запустить приложение
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```

Приложение будет доступно по адресу: **http://localhost:8000**

---

### Вариант 2 — через Docker Compose (рекомендуется)

```bash
# 1. Клонировать репозиторий
git clone <repo-url>
cd blog_api

# 2. Создать .env (единственный источник секретов — используется и compose, и приложением)
cp .env.example .env
# Обязательно сменить SECRET_KEY перед реальным/продакшен запуском

# 3. Запустить контейнер (сборка + старт)
docker compose up -d --build

# 4. Заполнить БД начальными данными (однократно)
docker compose exec api python seed.py

# 5. Остановка
docker compose down
```

> **Примечание:** файл базы данных SQLite хранится в Docker volume `db_data` и **не теряется** при перезапуске или пересборке контейнера.

### Эндпоинты

#### Аутентификация (`/api/auth`)

| Метод | URL              | Описание              | Доступ |
|-------|------------------|-----------------------|--------|
| POST  | `/auth/register` | Регистрация           | Все    |
| POST  | `/auth/login`    | Получение JWT-токена  | Все    |

**POST /auth/register** — тело запроса:
```json
{
  "username": "john_doe",
  "password": "secret123",
  "role": "Reader"
}
```

**POST /auth/login** — тело запроса:
```json
{
  "username": "john_doe",
  "password": "secret123"
}
```
Ответ:
```json
{
  "access_token": "<jwt>",
  "token_type": "bearer"
}
```

**Параметры листинга:**

```
GET /api/posts?page=1&limit=10&category=Технологии&status=published&sort_by=popular
```

| Параметр   | Тип     | Описание                                          | По умолчанию |
|------------|---------|---------------------------------------------------|--------------|
| `page`     | integer | Номер страницы (≥ 1)                              | 1            |
| `limit`    | integer | Записей на страницу (1–100)                       | 10           |
| `category` | string  | Фильтр по названию категории                      | —            |
| `status`   | string  | Фильтр по статусу (`draft` / `published`)         | —            |
| `sort_by`  | string  | Сортировка: `popular` (по лайкам) / `newest`      | newest       |

**Пример ответа:**
```json
{
  "total": 42,
  "page": 1,
  "limit": 10,
  "pages": 5,
  "items": [...]
}
```

#### История действий (`/api/logs`)

| Метод | URL          | Описание                                          | Доступ     |
|-------|--------------|----------------------------------------------------|------------|
| GET   | `/logs/`     | Постоянная история действий над сущностями          | Moderator  |

В отличие от вывода модуля `logging` (stdout, теряется при перезапуске контейнера без отдельного volume под лог-файлы), эта история хранится в таблице `action_logs` той же БД, что и сами сущности, и переживает перезапуск. Каждое создание/изменение/удаление статьи, комментария, категории, лайка или пользователя фиксируется с указанием: типа сущности, её ID, типа действия (`create`/`update`/`delete`), ID пользователя-инициатора и краткими деталями изменения.

Параметры:

| Параметр      | Тип     | Описание                                                   |
|---------------|---------|--------------------------------------------------------------|
| `page`        | integer | Номер страницы (≥ 1)                                          |
| `limit`       | integer | Записей на страницу (1–100)                                   |
| `entity_type` | string  | Фильтр: `User` / `Category` / `Post` / `Comment` / `Like`     |
| `action`      | string  | Фильтр: `create` / `update` / `delete`                        |
| `user_id`     | integer | Фильтр по ID пользователя-инициатора                          |

Пример:
```
GET /api/logs/?entity_type=Post&action=delete
```

> Данные, созданные скриптом `seed.py`, не попадают в эту историю — сидирование выполняется напрямую через модели SQLAlchemy, минуя CRUD-слой, поскольку у стартовых данных нет реального пользователя-инициатора. История начинает заполняться с первого реального действия через API.

## Валидация ключевых эндпоинтов

| Эндпоинт            | Правило                                                       | Статус |
|---------------------|---------------------------------------------------------------|--------|
| POST `/auth/register` | `username`: 3–64 символа, только `[a-zA-Z0-9_]`            | 422    |
| POST `/auth/register` | `password`: минимум 6 символов                              | 422    |
| POST `/auth/register` | `role` = `Moderator` (запрещено при самостоятельной регистрации) | 422    |
| POST `/auth/register` | `username` уже занят                                        | 400    |
| POST `/auth/login`  | Неверные логин / пароль                                       | 401    |
| Защищённые роуты    | Отсутствует или невалиден JWT                                 | 401    |
| Защищённые роуты    | Недостаточно прав (роль)                                      | 403    |
| GET `/posts/`       | `page` < 1 или `limit` вне диапазона 1–100                   | 422    |
| GET `/posts/`       | `sort_by` не `popular` и не `newest`                          | 422    |
| POST `/posts/`      | `title` пустой или > 256 символов                             | 422    |
| POST `/posts/`      | `content` пустой                                              | 422    |
| GET `/posts/{id}`   | Статья не найдена                                             | 404    |
| PUT `/posts/{id}`   | Author пытается изменить чужую статью                         | 403    |
| POST `/categories/` | Категория с таким именем уже существует                       | 400    |

Все ошибки возвращаются в виде JSON:
```json
{
  "detail": "Описание ошибки"
}
```

## Матрица ролей

| Действие                              | Reader | Author | Moderator |
|----------------------------------------|:------:|:------:|:---------:|
| Просмотр статей и комментариев        | +      | +      | +         |
| Добавление комментария                | +      | +      | +         |
| Лайк статьи                           | +      | +      | +         |
| Создание статьи                       | -      | +      | +         |
| Редактирование своей статьи           | -      | +      | +         |
| Удаление своей статьи                 | -      | +      | +         |
| Редактирование/удаление чужих статей  | -      | -      | +         |
| Редактирование/удаление чужих коммент.| -      | -      | +         |
| Управление категориями                | -      | -      | +         |
| Просмотр списка пользователей         | -      | -      | +         |
| Изменение роли пользователя           | -      | -      | +         |

> **Важно:** через `POST /auth/register` можно самостоятельно выбрать только роль `Reader` или `Author`. Роль `Moderator` нельзя получить регистрацией — её может выдать только уже существующий Moderator через `PATCH /users/{id}` (так нельзя выдать себе административные права в обход системы).

## Тестовые учётные данные (из seed.py)

| Роль       | Логин         | Пароль      |
|------------|---------------|-------------|
| Reader     | `reader_user` | `reader123` |
| Author     | `author_user` | `author123` |
| Moderator  | `moderator`   | `moder123`  |

**Быстрый старт через curl:**
```bash
# Получить токен
TOKEN=$(curl -s -X POST http://localhost:8000/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"author_user","password":"author123"}' | python -c "import sys,json; print(json.load(sys.stdin)['access_token'])")

# Создать статью
curl -X POST http://localhost:8000/api/posts/ \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"title":"Моя первая статья","content":"Привет, мир!","status":"published"}'
```
